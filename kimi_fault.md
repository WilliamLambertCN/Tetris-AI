# AI 开发反思记录

## 主要错误

### 1. 最初设计错误：禁用自然重力
**错误**：为了让 AI 精准控制，曾试图禁用游戏自然重力。
**后果**：AI 在作弊环境下运行，失去真实意义。
**纠正**：恢复自然重力，AI 必须在真实游戏环境下运行。

### 2. 疯狂插入 down 指令
**错误**：在动作队列中疯狂插入 `down` 指令，试图加速下落。
**后果**：
- 动作队列被 down 填满
- 水平移动指令被淹没
- 方块无法正常移动到目标位置
**纠正**：移除多余的 down，只保留基本动作（旋转、水平移动、硬降）。

### 3. 新方块检测逻辑反复出错
**错误**：多次修改新方块检测逻辑，但始终有问题：
- 使用 `piece_y <= 2` 判断，但检测条件太宽松
- 添加 `processing_piece` 标志但逻辑混乱
- 使用 `waiting_for_new_piece` 但时机不对
**后果**：
- 同一个方块被多次检测为新方块
- 动作队列被反复清空重建
- 方块堆积状态丢失（棋盘状态回退）
**纠正**：使用 `fast_dropping` 标志 + Y位置变化检测，只在硬降后检测新方块。

### 4. 棋盘状态不同步
**错误**：AI 根据前端上报的棋盘做决策，但前端执行动作有延迟。
**后果**：
- AI 看到的棋盘状态是过时的
- 决策基于错误状态
- 模拟放置的棋盘和实际不符
**反思**：AI 必须信任前端上报的状态，不能自己模拟预测。

### 5. while 循环 down 导致新方块被错误下降
**错误**：在 while 循环中发送 `down` 时，没有严格检测新方块已经出现。
**后果**：
- 新方块生成后，while 循环还在发送 `down`
- 新方块被错误地下落，导致堆积位置错误
- 棋盘状态混乱
**根因**：`hard_drop` 后方块已经落地并生成新方块，但我的 while 循环又发送了 `down`，影响到了新方块。
**纠正**：应该在检测到类型变化时 **立即** 退出 while 循环，不再发送任何 `down`。

### 6. 破坏用户写好的快速下降逻辑
**错误**：用户已经给出了正确方案："当方块位于 target 位置正上方时，while 循环调用 down，间隔 0.1s，直到新方块出现"。但我没有按这个方案实现，而是：
- 自作聪明地添加了 `fast_dropping` 标志
- 试图用复杂的状态机管理下降逻辑
- 反复修改引入新 bug，导致方块堆积状态丢失
**后果**：
- 同一个方块被重复检测为新方块
- 棋盘状态回退（之前放置的方块消失）
- 快速下降逻辑完全失效
**纠正**：应该严格按照用户的方案实现：
```python
# 用户方案（简洁可靠）
if piece_x == target_x:
    while not new_piece_detected:
        send_action(DOWN)
        sleep(0.1)
```

### 6. hard_drop 实现问题
**错误**：前端 `hard_drop` 使用 while 循环调用 `moveDown`，但游戏有自然重力干扰。
**后果**：
- 方块在自然重力和 hard_drop 双重作用下位置偏移
- 落点不准确
**纠正**：`hard_drop` 应该直接锁定方块到最终位置，而不是逐步下落。

### 7. 反复修改引入新 bug
**错误**：每次修改都试图修复一个问题，但引入新问题。
**表现**：
- 修复鬼畜 → 引入消除 bug
- 修复消除 → 引入检测 bug
- 修复检测 → 丢失快速下降
**反思**：应该一次性设计完整方案，而不是打补丁。

## 正确的设计原则

1. **AI 只能像人类一样操作**：通过左/右/旋转/下落控制，不能作弊
2. **信任前端状态**：AI 根据前端上报的棋盘做决策，不自己模拟
3. **简洁的动作队列**：只生成必要的动作，不插入多余的 down
4. **可靠的新方块检测**：使用类型变化 + 位置变化双重验证
5. **快速下降作为独立模式**：hard_drop 后进入快速下降，检测到新方块后退出

## 当前状态

AI 基本能运行，但仍有偶发的状态不同步问题。需要进一步测试验证。
